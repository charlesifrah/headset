MVP Spec
Working title

Shared Field

Core idea

Two players are connected to the same live simulation, but each sees a different reality.

Player A sees a garden and believes they are growing plants.

Player B sees a fire zone and believes they are putting out fires.

Both players:

move in the same map

perform the same underlying action

see each other in real time

influence the same hidden variables

receive different visual and textual interpretations

The point of the game is to prove that two people can share consequences without sharing the same perception.

Product goal

Build a simple multiplayer web/mobile prototype that demonstrates:

Two players can interact with the same system in real time

Each player can see the other player’s actions

Each player interprets those actions differently

Both players feel like they are playing the same game

A final reveal shows they were seeing different worlds

This MVP should be small, stable, and easy to understand in under 2 minutes of play.

Core gameplay loop
Round length

60 to 90 seconds

Player goal

Keep the field stable by responding to active zones faster and more efficiently than the other player.

What actually happens

The hidden simulation has zones with an internal value called instability.

Instability rises over time in random zones

If a zone crosses thresholds, it visually escalates

Players reduce instability by aiming and holding their spray action on that zone

Stabilized zones improve the player’s contribution score

Unchecked zones can spread instability to neighboring zones

What Player A believes

dry areas need water

watering grows plants

greener areas mean success

partner is also watering plants

score = plants grown / gardens saved

What Player B believes

burning areas need suppression

spraying puts out fires

smoke clearing means success

partner is also extinguishing fires

score = fires extinguished / sectors saved

Core design principle

There is only one game state.

There are two presentation layers.

Never create separate logic for each worldview.
Only translate the same shared state into two different visual and language systems.

This keeps the illusion clean and the code maintainable.

Hidden simulation model
Field

A field contains a set of zones.

Example:

9 zones in a 3x3 layout

each zone has a hidden instability value from 0 to 100

Zone thresholds

0–24: calm

25–49: emerging issue

50–74: active problem

75–100: critical problem

Zone behavior

Every tick:

instability increases slightly in random zones

neighboring zones may receive spillover if a zone is above critical threshold

player action reduces instability while applied

Hidden meaning

This is the only truth the server cares about:

zone instability

player proximity

action uptime

contribution values

round timer

Everything else is client interpretation.

Map hidden instability to garden visuals:

0–24 → healthy soil / blooming plants

25–49 → slightly dry / drooping leaves

50–74 → wilting patch / brown edges

75–100 → severely dry / near-dead growth

Action effect:

spray = watering

instability reduction = hydration

spillover = nearby dryness spreading

Score labels:

Plants Grown

Garden Saves

Water Efficiency

Player B, Fire mode

Map hidden instability to fire visuals:

0–24 → safe ground / cooled area

25–49 → embers / light smoke

50–74 → active flames

75–100 → severe blaze / heavy smoke

Action effect:

spray = extinguishing

instability reduction = suppression

Score labels:

Multiplayer behavior
Critical requirement

Each player must see the other player acting in real time.

But the partner is always translated into the viewer’s own reality.

Example

Player B sprays a critical zone.

B sees: foam or water extinguishing flames

A sees: partner watering a withered patch and restoring growth

Both are looking at the same event:

same coordinates

same duration

same server effect

different interpretation

This is the core psychological mechanic.

Mobile

That reduces control complexity and speeds up development.

Core mechanics in detail
1. Zone escalation

The server periodically increases instability in zones.

Pseudo behavior:

every 500ms, choose 1 to 3 zones

add +2 to +8 instability

cap at 100

2. Player action

When player is:

close enough to a zone

aiming at that zone

holding action

Then server applies:

instability reduction = X per tick

Example:

reduce by 8 per 250ms tick

3. Spread

If a zone remains above 75 too long:

after N seconds, nearby zones gain +10 to +20 instability

This creates urgency and visible chain reactions.

4. Contribution scoring

Each time a player reduces instability:

assign contribution points proportional to reduction

Each time a zone returns below a key threshold:

award save / recovery points to the players who contributed recently

This lets both players “compete” while still co-influencing the same field.

Scoring model

Use one hidden score model, translated differently per player.

Hidden score metrics

total instability reduced

number of critical saves

average response time

active action time

chain prevention count

Garden labels

Plants Grown = total instability reduced

Garden Saves = critical saves

Water Efficiency = total reduction / action time

Bloom Chains = chain prevention count

Fire labels

Fires Extinguished = total instability reduced

Sector Saves = critical saves

Suppression Efficiency = total reduction / action time

Flare Stops = chain prevention count

The numbers can be identical. Only the labels differ.

UX rules
Rule 1

Both players must feel like the game is coherent in their own view.

Rule 2

Neither player should suspect immediately that the other sees a different world.

Rule 3

All partner actions must make sense in the local player’s language.

Rule 4

The reveal must clearly map one reality to the other.

If the reveal is weak, the whole concept weakens.

Shared simulation

Player A sees garden assets and labels only

Risks and mitigation
Risk

Players immediately realize the trick.

Fix

Keep vocabulary parallel and avoid obvious contradiction during gameplay.

Risk

Multiplayer sync feels laggy.

Fix

Server authoritative state with client interpolation, keep map small.

Risk

Too much control complexity on mobile.

Fix

Use simplified tap-to-zone or nearest-zone action for MVP.

Risk

Concept feels like only a visual reskin.

Fix

Make the reveal explicitly show that all scores and events were shared underneath.

Best ultra-simple MVP variant

If you want the fastest possible build, make it even simpler:

fixed 3x3 grid

players do not free-move

each player taps zones directly

hold to apply action

same zone state underneath

live cursor indicators show where the partner is acting

each player sees their own translated effects

end reveal maps all actions

This version can still prove the idea with far less engineering.

Suggested first build target

Build this first:

A 2-player browser prototype with a 3x3 grid, 60-second round, tap-and-hold action, garden/fire perception split, live partner indicators, and a reveal screen.

That is the fastest route to proving the concept.
